Context

Native Base is a pure HTML/CSS/JS example repository for 2023, aiming for best-practice patterns and to be used as a starting place upon which to build other projects. View the source code. Learn more about this project's goals on GitHub.

Code style

  • Tabs for indentation
  • Spaces for alignment

Tabs should be used for indentation, because:

  • Screen reader technology such as Braille displays handle tabbed indentation better than spaces used for indentation.
  • Tab width is configurable by the person looking at the code, who may have vision issues. If they prefer a tab to be 2, or 4, or 8 spaces wide they can set that in their editor. Just as they can set line heights. So can you.
  • Using tabs is being respectful of other people. It does not stop your own editor from displaying exactly how you want it, and it allows others to have it how they want it.

I'm pretty strict about my indentation, because I use code-folding heavily. Code folding allows me to collapse sections of code that are not relevant to the current task, which removes them from my mind so I can concentrate on what I'm working on without visual clutter.

Code comments

I try to comment liberally.

When a server sends HTML it should be configured to minify and compress the mark-up, which removes HTML comments too. Code comments have no impact on the end product, and liberally using them will help others - and yourself - understand what you were thinking, referencing, and doing at the time you wrote the code.

If you're working in a templating language, it may make sense to use the template languages comments rather than HTML's more verbose comment syntax.
