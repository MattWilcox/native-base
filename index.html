Context

Native Base is a pure HTML/CSS/JS "starter" system for 2023, aiming for best-practice patterns as a starting place upon which to build other projects. Learn more about this project's goals on GitHub.

Coding Style

  • Tabs for indentation
  • Spaces for alignment

There is no argument. Tabs should always be used for indentation and it is not a matter of author preference, it is a matter of respecting other people and allowing them to have their own accommodations.

Tab width is configurable by the person looking at the code. If they prefer a tab to be 2, or 4, or 8 spaces wide they can set that in their editor. Just as they can set line heights. So can you. Further, screen reader technology handles tabbed indentation better than spaces used for indentation.

Be respectful of other people. Use tabs. It does not stop your own editor from displaying exactly how you want it, and it allows others to have it how they want it.

I will not put up with people arguing that tabbed indentation is somehow an inconvenience to them because tabs are too deep for their preference, and they are used to typing two spaces or something. Learn your editor, configure that depth to your liking. If you can program, you can work that out. Use the tab key, use the tab character.

Virtually any other coding style I'm flexible on.

Comments

When a server sends HTML it should ideally be configured to minify and compress the mark-up, which removes HTML comments too. As such; liberally comment your code. It has no impact on the end product, and it will help others - and yourself - understand what you were thinking, referencing, and doing at the time.

If you're working in a templating language, it may make sense to use the template languages comments rather than HTML's more verbose comment syntax.
