Tables

Tables are a tricky from a practical perspective as they don't work well in confined spaces. Tables by definition use two-dimensionality as a tool of semantics and presentation. So they are terrible when the screen they are being shown on is too small to fit the table into - one of your dimensions ends up too small to fit in the data.

Tables tend to work well only on larger screens, and so you should make sure you have a good reason for displaying tabular data, and prefer other solutions if practical.

There are methods for adapting tabular data for display on smaller screens, but be aware none are without compromise:

  • If you set display: block; on any table element, you will strip it of the native accessibility enhancements it has, leading to a loss of affordances for people using assistive devices.
  • If you rely on over-scroll, it becomes hard to have an overview of the table at a glance, and less clear.

There is no "right choice".

Form markup guide

Example Form

This shows my standardised method for marking up form content.

Form Instructions

This is where instructions for filling out the form can be added. For example:

  • Remember to gain the consent of a parent or guardian before completing this form.

Form Errors

This is where errors with the form can be output.

For example, individual fieldsets may be valid, but the combination of values between fieldsets may not be considered valid:

  • "Guardian Details > Address" should be the same as "Child Details > Address".

Fieldset Instructions

This is where instructions for filling out the fieldset can be added. For example:

  • When completing this field-group, remember the details should be for how old you will be when the event starts, and not how old you are today.

Fieldset Error

This is where errors with the fieldset can be output.

For example, the individual fields may be valid, but the combination of values may not be considered valid:

  • The "Start Date" should be earlier than the "End Date".
